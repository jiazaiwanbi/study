### range channel 的特殊性

在 Go 中，使用 `for range` 循环读取 channel 时，这个循环会一直运行，直到遇到以下两种情况之一：

1. channel 被关闭（close）
2. channel 被垃圾回收

---

### 如果不 close 会怎样？

让我们修改代码，去掉 close 看看会发生什么：

```go
package main

import (
    "fmt"
)

func main() {
    ch := make(chan int)
    go func() {
        for i := 0; i < 5; i++ {
            ch <- i
        }
        // 注释掉 close(ch)
    }()

    for v := range ch {
        fmt.Println(v)
    }
    fmt.Println("结束") // 这一行永远不会执行到
}
```

运行这段代码，你会发现：

1. 前 5 个数字正常打印
2. 程序不会结束，而是卡在那里
3. 发生了死锁（deadlock）

---

### 为什么会死锁？

这是因为：

1. `for range` 循环在等待 channel 中的数据
2. channel 没有被关闭，所以 `for range` 认为还可能有数据到来
3. 但是发送数据的 goroutine 已经结束了，不会再发送数据
4. 这就造成了主 goroutine 永远在等待，形成死锁

---

### 正确的做法

所以正确的做法是：

1. 当我们确定不再向 channel 发送数据时，应该显式地关闭它
2. 这样 `for range` 循环就知道不会再有新数据了，可以安全退出

```go
ch := make(chan int)
go func() {
    for i := 0; i < 5; i++ {
        ch <- i
    }
    close(ch)  // 明确告诉接收方：不会再有数据了
}()

for v := range ch {
    fmt.Println(v)
}
fmt.Println("现在程序可以正常结束了")
```

---

### 实用小贴士

1. 在实际开发中，通常由发送方负责关闭 channel
2. 接收方不应该关闭 channel，因为它不知道发送方是否还会发送数据
3. channel 的关闭是一个信号机制，表示"不会再有数据了"
4. 关闭后的 channel 仍然可以接收数据（接收到零值），但不能发送数据

---

### 小练习

试试修改上面的代码：

1. 注释掉 `close(ch)` 看看会发生什么
2. 尝试向已关闭的 channel 发送数据，看看会发生什么
3. 尝试从已关闭的 channel 接收数据，观察接收到的值

这样的实践可以帮助你更好地理解 channel 的行为！
